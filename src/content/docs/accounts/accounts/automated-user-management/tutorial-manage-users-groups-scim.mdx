This tutorial will walk you through some common procedures for managing users on the New Relic One user model using the SCIM API. The SCIM API allows you to view, create, update, and delete users and groups programmatically, outside of the [User management UI](#TODO).

## Requirements [#requirements]

Some notes on who this tutorial is for and requirements:

* This tutorial is for managing users on our New Relic One user model.
* This tutorial is for managing users without a supported identity provider. Refer to one of the specific integration guides to manage users with a supported identity provider: [Okta](/docs/accounts/accounts/automated-user-management/okta-scimsso-application-configuration/), [Azure AD](/docs/accounts/accounts/automated-user-management/azure-ad-scimsso-application-configuration/), [OneLogin](/docs/accounts/accounts/automated-user-management/azure-ad-scimsso-application-configuration/)
* Managing users and groups with the SCIM API requires Enterprise edition. For details, see [user management requirements](/docs/accounts/accounts-billing/new-relic-one-user-management/add-manage-users-groups-roles#requirements).
* Note that managing _access grants_ is not possible with the SCIM API, you must use the **Organization and access** UI to create grants. For details, see [user management tutorial](#TODO).
* [Role](/docs/accounts/accounts-billing/new-relic-one-user-management/new-relic-one-user-model#standard-roles) requirements:
    * Generating the SCIM API token requires the **Authentication domain manager** role.
    * Using the **Organization and access** UI to manage group access requires **Authentication domain manager** and **Organization manager** roles.

## Configure your Authentication Domain for SCIM [#configure-scim]

Before you can use the SCIM API, you must first [enable SCIM for your authentication domain](/docs/accounts/accounts-billing/new-relic-one-user-management/authentication-domains-saml-sso-scim-more/#source-users). Note the API access token will be shown only once after saving the configuration, so keep it someplace safe! 

<Callout variant="tip">
If you need to view a bearer token later, the only way to do this is to generate a new one, and that will invalidate the old one and any integrations using the old token.
</Callout>

## Create users and user groups in your system [#create-in-idp]

The SCIM API is typically used for importing users and groups into New Relic from a third-party identity provider. Create users and groups in your system before proceeding in this tutorial.

If you want to use the SCIM API in a custom application or for ad-hoc requests, proceed to learn how to connect to the SCIM API.

## Connect to the SCIM API [#connect]

The SCIM API is available at `https://scim-provisioning.service.newrelic.com/scim/v2` and this URL is viewable in the Authentication Domain settings page. To access the SCIM API, your client must include a **bearer token** with each request. The token is displayed after saving your Authentication Domain configuration.

If you are using a third-party identity provider, configure it to use **Bearer Token authorization** and plug in your API access token. Refer to your identity provider's documentation for help configuring this. Once configured, you're all set to import users and groups!

The SCIM 2.0 protocol requires user and group data to be sent in accordance with a few core _schemas_ and encoded in JSON. See [RFC 7643 - SCIM Core Resources and Extensions](https://datatracker.ietf.org/doc/html/rfc7643#section-4) and [RFC 7643 - JSON Representation](https://datatracker.ietf.org/doc/html/rfc7643#section-8) for the specifics. Refer to [RFC 7644 - SCIM Protocol](https://datatracker.ietf.org/doc/html/rfc7644) for more information about the protocol used in this
tutorial.

For all requests to the SCIM API, you must provide the bearer token in an `Authorization` header. Here's an example with `curl`:

```
curl -X 'GET' -H 'Accept: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Users'
```

Any request in the rest of this tutorial will receive a **401 Unauthorized** response if the API access token is missing or invalid.

Example response:

<Collapser title="401 Unauthorized">
  ```
  {
    "schemas": [
      "urn:ietf:params:scim:api:messages:2.0:Error"
    ],
    "detail": "invalid authentication token",
    "status": "401"
  }
  ```
</Collapser>

## Create users in your Authentication Domain [#create-users]

If you are using a third-party identity provider, refer to their documentation for how to push users with SCIM.

Otherwise, send a `POST` request to `/scim/v2/Users` to create a user. The following user attributes are **required**:
* `userName` This identifier **must** be unique within your Authentication Domain. Use the user's email address. 
* `emails` Email address of the user.
* `active` Boolean indicating whether or not the user should be active or inactive within New Relic.

It is recomended to provide the following attributes for the best user experience:
* `name.givenName` The user's first or given name.
* `name.familyName` The user's last or family name.
* `timezone` The user's timezone in IANA Time Zone database format.

Take note of the returned user `id`. To update a user in the future you will need to supply the same ID with the request.

```
curl -X 'POST' -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Users' --data-binary @- <<EOF
{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
  "userName": "bjensen@example.com",
  "name": {
    "familyName": "Jensen",
    "givenName": "Barbara"
  },
  "emails": [
    {
      "primary": true,
      "value": "bjensen@example.com"
    }
  ],
  "active": true,
  "timezone": "America/Los_Angeles"
}
EOF
```

## Create groups in your Authentication Domain [#create-groups]

If you are using a third-party identity provider, refer to their documentation for how to push groups with SCIM.

Otherwise, send a `POST` request to `/scim/v2/Groups` to create a group. The only group attribute **required** is:
* `displayName` The group name.

Take note of the returned group `id`. To update a group or its members in the future you will need to supply the same ID with the request.

```
curl -X 'POST' -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Groups' --data-binary @- <<EOF
{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
    "displayName": "Example Group"
}
EOF

## View users and groups in your Authentication Domain [#view]

After you've created some users and groups you will see them in the [User management UI](#TODO). You can also retrieve them from the SCIM API.

In this tutorial we will search for specific users and groups, but that is not the only way to view users and groups. Refer to the [SCIM  API reference](/docs/accounts/accounts/automated-user-management/scim-support-automated-user-management/) and [RFC 7644](https://datatracker.ietf.org/doc/html/rfc7644) for all the available query options.

To retrieve a user by email, send a `GET` request to `/scim/v2/Users` with a `filter` query parameter. The `filter` parameter must be URL encoded.

```
curl -X 'GET' -H 'Accept: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Users' --get --data-urlencode 'filter=userName eq "bjensen@example.com"'
```

## Update a user's attributes [#update-user]

The SCIM API supports both `PUT` and `PATCH` methods for updating users. Refer to the [SCIM API supported actions](/docs/accounts/accounts/automated-user-management/scim-support-automated-user-management/#supported-actions) and [RFC 7644](https://datatracker.ietf.org/doc/html/rfc7644) for details on using `PATCH`. This tutorial will demonstrate updating a user's attributes with the `PUT` method.

New Relic does **not** require all user attributes be included in the request body, only the attributes you want to update are necessarry. Send a `PUT` request to `/scim/v2/Users/${ID}` to update the user.

```
curl -X 'PUT' -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Users/5a1d580f-323c-450c-8c62-479b5c9085d6' --data-binary @- <<EOF
{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
  "timezone": "America/Chicago"
}
EOF
```

## Update a group's members [#update-group]

The SCIM API supports both `PUT` and `PATCH` methods for updating groups. This tutorial will demonstrate updating a group's members with the `PATCH` method. Refer to the [SCIM API supported actions](/docs/accounts/accounts/automated-user-management/scim-support-automated-user-management/#supported-actions) and [RFC 7644](https://datatracker.ietf.org/doc/html/rfc7644) for details on using `PUT`.

`PATCH` is convenient for adding or removing group members without needing to specify the full member list in the request. To add a user to a group, use the following operation parameters:
* `op` set to `Add`
* `path` set to `members`
* `value` set to a list of `{"value": "${USER_ID}"}` with each user ID to add to the group

Send a `PATCH` request to `/scim/v2/Groups/${ID}` to update group members.

```
curl -X 'PATCH' -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Groups/df2b9a04-0426-4a3e-bf5f-54d5341f4e5b' --data-binary @- <<EOF
{
  "schemas": [
    "urn:ietf:params:scim:api:messages:2.0:PatchOp"
  ],
  "Operations": [{
    "op": "Add",
    "path": "members",
    "value": [{
      "value": "5a1d580f-323c-450c-8c62-479b5c9085d6"
    }]
  }]
}
EOF
```

Example response:

<Collapser title="200 OK">
```
{
  "schemas": [
    "urn:ietf:params:scim:schemas:core:2.0:Group"
  ],
  "id": "df2b9a04-0426-4a3e-bf5f-54d5341f4e5b",
  "displayName": "Example Group",
  "meta": {
    "resourceType": "Group",
    "created": "2021-07-20T22:41:50.414Z",
    "lastModified": "2021-07-20T22:41:50.414Z"
  },
  "members": [
    {
      "type": "User",
      "value": "5a1d580f-323c-450c-8c62-479b5c9085d6"
    }
  ]
}
```
</Collapser>

And to remove a user from a group, use the following operation parameters:
* `op` set to `Remove`
* `path` set to `members`
* `value` set to a list of `{"value": "${USER_ID}"}` with each user ID to remove from the group

```
curl -X 'PATCH' -H 'Accept: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Groups/df2b9a04-0426-4a3e-bf5f-54d5341f4e5b' --data-binary @- <<EOF
{
  "schemas": [
    "urn:ietf:params:scim:api:messages:2.0:PatchOp"
  ],
  "Operations": [{
    "op": "Remove",
    "path": "members",
    "value": [{
      "value": "5a1d580f-323c-450c-8c62-479b5c9085d6"
    }]
  }]
}
EOF
```

Example response:

<Collapser title="200 OK">
```
{
  "schemas": [
    "urn:ietf:params:scim:schemas:core:2.0:Group"

## Delete users and groups [#delete]

To remove a user from your Authentication Domain, send a `DELETE` request to `/scim/v2/Users/${ID}`.

```
curl -X 'DELETE' -H 'Accept: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Users/d0f4d8e3-5413-4894-a8f9-de709994e18c'
```

Example response:

<Collapser title="204 No Content">
</Collapser>

Similarly, to remove a group from your Authentication Domain, send a `DELETE` request to `/scim/v2/Groups/${ID}`.

```
curl -X 'DELETE' -H 'Accept: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Groups/df2b9a04-0426-4a3e-bf5f-54d5341f4e5b'
```

Example response:

<Collapser title="204 No Content">
</Collapser>

## Optional: Manage user type with SCIM

By default, a user's type can only be viewed and modified in the [User management UI](#TODO). If you'd like to use SCIM instead, update your Authentication Domain configuration to [Delegate control of user type](#TODO) to your identity provider or custom application.

The user's type attribute is defined in the custom schema `urn:ietf:params:scim:schemas:extension:newrelic:2.0:User`. Include this schema and the `nrUserType` string attribute in your create or update request to set a user's type.

Valid values for `nrUserType` include:
* `Full User`
* `Basic User`

To create a new `Basic User` send a `POST` request `/scim/v2/Users` and include the custom New Relic schema extension:

```
curl -X 'POST' -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" 'https://scim-provisioning.service.newrelic.com/scim/v2/Users' --data-binary @- <<EOF
{
    "schemas": [
      "urn:ietf:params:scim:schemas:core:2.0:User",
      "urn:ietf:params:scim:schemas:extension:newrelic:2.0:User"
    ],
    "userName": "jbenson@example.com",
    "name": {
        "givenName": "James",
        "familyName": "Benson"
    },
    "emails": [{
        "primary": true,
        "value": "jbenson@example.com",
        "type": "work"
    }],
    "active": true,
    "timezone": "America/Chicago",
    "urn:ietf:params:scim:schemas:extension:newrelic:2.0:User": {
      "nrUserType": "Basic User"
    }
}
EOF
```
